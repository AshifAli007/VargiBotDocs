node_ur5_1_pick_and_place_controller module¶

This node is controlling ur5_1 and acts as a Action Server

It also uses the 2d camera to detect all the packages and seperates them in arrays according to their priority and use these arrays while processing goals/orders. It uses 2d camera info to update inventory sheet of IMS.

goal format ->
String priority String order # Contain all the details in form of string of dict

Flow:- As soon as goal/order is recieved by this node it seperates them in arrays according to their priority and starts a seperate thread[orders_handler] to process current and all the upcoming goals. main thread continue to recieve goals and order_handler thread process the goal which are present in arrays according to their priority. As soon as the processing is completed it updates the ordersDispatched sheet of IMS.

callback(data)¶

This Callback is responsible for mapping packages with their color.

for mapping the colors we first take the image from camera1 and than crop this image so that only region of interest is present in cropped image than we use this cropped image to get qr_data.

# Ref: # https://stackoverflow.com/questions/9084609/how-to-copy-a-image-region-using-opencv-in-python # https://stackoverflow.com/questions/15589517/how-to-crop-an-image-in-opencv-using-python

for eg, if we want the color of packagen00 we first take the image, now we know that packagen00 lies in a square made with points (100,300)[top left vertex](250,450)[bottom right vertex] so this is the region of interest for packagen00 now we crop this the image to get only region of interest [ROI] and send this cropped image with only packagen00 to function get_qr_data() which gives us the color of package00 now we map this color with box_name and details in a global lists according to their priority and using these lists while processing goal.

on_goal(goal_handle)¶

As soon as the Action client send a goal to the server this

functions is called which is responsible for seperating the goals accroding to their priorty in lists this list is than processed in a seperate thread called order_handler which is called only once when the very first order is recieved by the server.

Parameters:goal_handle – contain the details of order.
populate_orders_dispatched(priority, order)¶

As soon as goal/order processing is completed this

function is called to update ordersDispatched sheet of IMS. As the sequence in which incoming orders are coming is not same as the sequence in which orders are processing that’s why we can’t use formulae for updating some cells of Dashboard because these cells are dynamic and may change due to different priority that’s why this function also update some cells of Dahshboard by finding required cell using order_id.

Parameters:
  • priority – (str)
  • order – (dict) details of order.
process_goal()¶

This function is responsible for

Handeling all the goals present in global lists according to their priority so high priority orders are processed first and if no goal is present in high priority list it than checks for medium than low priority orders iteratively till it finds an order to process.

publish_picked_order(priority, order)¶

As soon as an order is picked for processing

this order is also published to a topic which is subscribed by node_ur5_2_controller to save the sequence in which orders are processed, thus during shipping node_ur5_2 uses this saved sequence to update ordersShipped sheet of IMS.

Parameters:
  • priority – (str)
  • order – (str) string of dictionary.
